# 왜 온톨로지인가 — 검색·LLM 단독과의 실측 비교

콘크리트·안전 검측 질문을 두 방식에 돌려 차이를 측정한 기록이다.

- **LLM 단독 / 검색기반(RAG)**: 법령·기준 텍스트를 찾아(또는 모델 기억으로) 자연어로 답한다.
- **온톨로지 엔진(이 저장소)**: 측정값을 결정적 규칙으로 계산해 합/부를 판정하고 근거 조항(§)을 인용한다.

결론을 먼저 말하면 — **한도가 문장에 숫자로 적힌 단순 조회는 검색·LLM도 맞힌다.** 갈리는 곳은
(B) 한도를 *계산*해야 하거나, (C) 범주에 따라 한도가 *달라지거나*, (D) *시점 게이트*가 걸리는 질문이다.
거기서 텍스트 방식은 그럴듯하게 빗나가고, 엔진은 같은 입력에 항상 같은 판정 + 개정일·출처를 낸다.

재현: [`docs/why_ontology_demo.py`](./why_ontology_demo.py) — `python docs/why_ontology_demo.py` (키 불필요).

## 네 범주의 검측 질문

엔진 판정은 위 스크립트로 그대로 재현된다(8/8 결정적). 텍스트 방식 열은 동일 질문을 검색기반
RAG 및 범용 LLM 단독에 돌린 관측이다.

### A. 단일 임계값 조회 — 한도가 문장에 숫자로 있고, 판정 = 단순 비교

| 질문 (측정값) | 엔진 | 텍스트 방식 | 일치 |
|---|---|---|---|
| AE 공기량 **2%** | ✗ 부적합 `2.0 < 3.0` (KS F 4009) | ✗ 부적합 | ✅ |
| 염화물 **0.45** | ✗ 부적합 `0.45 > 0.30` (KS F 4009) | ✗ 부적합 | ✅ |
| EC4 / W/B **0.55** | ✗ 부적합 `0.55 > 0.45` (KDS 14 20 40) | ✗ 부적합 | ✅ |

→ 이 구간은 텍스트 방식도 맞힌다. 한도가 청크에 있고 비교가 한 줄이라서.

### B. 계산·유도 임계값 — 한도 = `0.85 × fck`, 문장에 숫자로 없음

| 질문 | 엔진 | 텍스트 방식 | 정오 |
|---|---|---|---|
| fck 24 / 시험 **19** | ✗ 부적합 `19 < 0.85×24 = 20.4` | "19<24 미달 → 불합격" | 결론만 맞고 **근거 오류**(0.85 미적용) |
| fck 24 / 시험 **21** | **✓ 합격** `21 ≥ 20.4` | **"21<24 미달 → 불합격"** | ❌ **거짓 불합격** |

→ 결정적 분기점. 텍스트 방식은 `0.85×fck`를 계산하지 못해 naive `21<24`로 빠지고,
**기준을 통과한 배치를 불합격으로 오판**한다. 합격 한도가 코퍼스에 숫자로 없기 때문이다.

### C. 범주별 한도 — 같은 값, 노출등급에 따라 한도가 달라짐

| 질문 | 엔진 | 텍스트 방식 |
|---|---|---|
| **EC2** / W/B 0.55 | ✓ 합격 `0.55 ≤ 0.55` (EC2 상한) | 표 회수 실패 시 "판단 불가" |
| **EC4** / W/B 0.55 | ✗ 부적합 `0.55 > 0.45` (EC4 상한) | (표 회수 성공 시) ✗ 부적합 |

→ 같은 0.55인데 등급만 다르다. 엔진은 양쪽 결정적. 텍스트 방식은 검색 결과에 따라
한쪽은 답하고 한쪽은 회피하는 비결정성이 나타난다.

### D. 시점 게이트 — 사고 발생일 vs 법 시행일

| 질문 | 엔진 | 텍스트 방식 |
|---|---|---|
| **2021** 사망사고, 중대재해처벌법 위반? | `not_applicable` — 2022-01-27 **시행 전, 소급 불가** | "정보 부족 — 판단 불가" |
| **2024** 사망사고(체계 미이행), 위반? | `violated` — §4/§9 | 시행일 축이 없어 판정 불가 |

→ 책임 귀속은 "사고 당시 그 법이 유효했나"에 달렸다. 엔진은 시행일을 1급 축으로 들고
게이트를 친다. 텍스트 방식에는 시간 축 자체가 없다.

## LLM 단독의 실패 모드 — 정확도가 아니라 신뢰

범용 LLM은 안정적·널리 알려진 값(예: 0.85fck, 중대재해처벌법 시행일 2022-01-27)은 잘 맞힌다.
실측에서 무너진 곳은 일정했다.

1. **시점 드리프트** — 개정된 기준을 *바뀐 줄 모르고* 옛 값으로 답한다.
   예: 굴착 안전기울기는 2021-11-19 별표11이 4분류로 재편됐는데, 구(舊) "보통흙 건지/습지" 분류로 답한다.
2. **프레임 혼동** — 같은 항목에 복수 기준 체계가 있을 때 다른 축의 값을 섞는다.
   예: 피복두께를 부재기반(기초·기둥·슬래브) 대신 노출기반(흙접촉·수중) 값으로 답한다 — 둘 다 실재해서 더 안 걸린다.
3. **플래그 부재** — 위 두 경우 모두 모델은 *확신 있게, 출처·개정일 없이* 답한다.
   엔진은 같은 규칙에 `revision: 2021-11-19` 와 `pending: [원문 대조 필요]` 를 들고 있어
   "이건 개정됐다 / 이건 미검증"을 구분한다.

## 정리

| 기준 | 검색·LLM 단독 | 온톨로지 엔진 |
|---|---|---|
| 단일 임계값 정확도 | 자주 맞음 | 항상 |
| 계산·범주·시점 | 거짓 판정·회피·불가 | 결정적 판정 |
| 같은 입력 → 같은 결과(보장) | ✗ | ✅ |
| 검증 가능 출처(§·개정일) | ✗ | ✅ |
| 모르는 것을 표시 | ✗ (그럴듯하게 채움) | ✅ `not_applicable` / `pending` |

검측·책임추적 맥락에서 *"가끔 맞는 그럴듯한 답"* 은 실격이다. 기준을 통과한 배치를 한 번이라도
거짓 불합격하면 불필요한 철거·재시공·분쟁으로 이어지고, 시행일을 한 번이라도 틀리면 책임 귀속이
어긋난다. 그래서 이 저장소는 **LLM이 판정하는 것을 구조적으로 막고**(자연어 슬롯 채우기·결과
서술만 허용), 판정은 항상 결정적 엔진이 한다. 관련 설계는 [`../rag/README.md`](../rag/README.md) 참조.
